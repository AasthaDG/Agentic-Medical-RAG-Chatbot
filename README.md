## 📌 Overview

This project implements a **production-grade agentic RAG pipeline** for medical question answering. Rather than relying on an LLM's parametric memory alone, the system retrieves semantically relevant doctor–patient conversations from a FAISS vector store and grounds every response in real clinical context.

Instead of one monolithic LLM call, the system uses **three specialized agents** — a planner, a retriever, and a generator — each with a distinct responsibility. This separation makes the system modular, debuggable, and measurably more accurate than a vanilla GPT baseline.

**Benchmark results (20 standardized medical queries):**

| Metric | Baseline (No RAG) | RAG System | Δ |
|---|---|---|---|
| Avg relevance score | 2.9 / 5.0 | 4.1 / 5.0 | **+41%** |
| Irrelevant outputs (score ≤ 2) | 10 / 20 | 6 / 20 | **−40%** |
| Retrieval latency | — | < 15ms | — |

> Relevance scored 1–5 by GPT-as-judge on groundedness and query relevance. See [`evaluate.py`](evaluate.py) for methodology.

## 🏗️ Architecture

```
User Query
    │
    ▼
┌──────────────────────────────────────┐
│            PlannerAgent              │
│  • Classifies intent (6 categories)  │  GPT-3.5  temp=0.1
│  • Rewrites query for clinical search│
│  • Flags medical emergencies         │
└──────────────────┬───────────────────┘
                   │  optimized search query
                   ▼
┌──────────────────────────────────────┐
│           RetrieverAgent             │
│  • Embeds query via MiniLM-L6-v2     │  No LLM — pure FAISS
│  • Cosine search over 10K+ docs      │
│  • Optional MMR for diversity        │
└──────────────────┬───────────────────┘
                   │  top-k context documents
                   ▼
┌──────────────────────────────────────┐
│           GeneratorAgent             │
│  • Prompt-engineered grounded answer │  GPT-3.5  temp=0.3
│  • Source citations returned         │
│  • Emergency routing built-in        │
└──────────────────┬───────────────────┘
                   │
                   ▼
        Structured Response
  { response, sources, intent, is_emergency }
```

**Key design principle:** The retriever makes zero LLM calls — only the planner and generator do. This keeps retrieval fast (< 15ms) and cheap.

## 🚀 Quick Start

### Prerequisites
- Python 3.10+
- OpenAI API key
- ~2GB disk space for the FAISS index

### 1. Clone & install

```bash
git clone https://github.com/yourusername/medical-rag-chatbot.git
cd medical-rag-chatbot
pip install -r requirements.txt
```

### 2. Configure environment

```bash
cp .env.example .env
```

Open `.env` and add your OpenAI API key:
```
OPENAI_API_KEY=sk-your-key-here
```

### 3. Build the vector index

Downloads 10K real doctor–patient conversations from HuggingFace, embeds them using `all-MiniLM-L6-v2`, and stores them in a FAISS index. **Run once** — takes ~5–10 minutes.

```bash
python ingest.py
```

You should see:
```
✅ Loaded 10,000 healthcare conversations
✅ Embeddings shape: (10000, 384)
✅ FAISS index built with 10,000 vectors
✅ Index and documents saved to disk
```

### 4. Launch the chatbot

```bash
python app.py
# → http://localhost:7860
```

For a **public shareable link** (great for portfolio):
```bash
python app.py --share
# → https://abc123.gradio.live
```

### 5. Run the benchmark

```bash
python evaluate.py
# → evaluation/benchmark_results.csv
# → evaluation/benchmark_chart.png
```

## 🔬 Technical Details

### Dataset

| Property | Value |
|---|---|
| Source | [ChatDoctor-HealthCareMagic-100k](https://huggingface.co/datasets/lavita/ChatDoctor-HealthCareMagic-100k) |
| Size used | 10,000 conversations (sampled, quality-filtered) |
| Format | `Patient: <question>\nDoctor: <answer>` |
| Quality filter | Input ≥ 20 chars, output ≥ 50 chars |

### Embedding Model

| Property | Value |
|---|---|
| Model | `sentence-transformers/all-MiniLM-L6-v2` |
| Embedding dim | 384 |
| Similarity metric | Cosine (L2-normalized inner product) |
| Inference | Local CPU — zero API cost per query |

### Vector Index

| Property | Value |
|---|---|
| Library | FAISS |
| Index type | `IndexFlatIP` — exact search, no approximation |
| Scale | 10K vectors (< 50MB on disk) |
| Query latency | < 15ms on CPU |

### Agent Design

| Agent | Responsibility | LLM? | Temp |
|---|---|---|---|
| `PlannerAgent` | Intent classification + query rewriting | ✅ GPT-3.5 | 0.1 |
| `RetrieverAgent` | FAISS semantic search + optional MMR | ❌ | — |
| `GeneratorAgent` | Grounded response generation + citations | ✅ GPT-3.5 | 0.3 |

### Intent Categories

The PlannerAgent classifies every query into one of:
`symptom_inquiry` · `medication_question` · `diagnosis_help` · `treatment_options` · `general_health` · `emergency`

Emergency queries immediately surface a 911 routing message before the generated response.

### Prompt Engineering

- **PlannerAgent** uses a structured output schema (`INTENT / SEARCH_QUERY / EMERGENCY`) at `temp=0.1` for deterministic classification
- **GeneratorAgent** uses a safety-anchored system prompt that instructs the model to stay grounded in retrieved references and flag when context is insufficient
- Temperature is intentionally different per agent — low for classification, slightly higher for generation

## 📊 Evaluation

Benchmark methodology: 20 standardized medical queries spanning all 6 intent categories. Each query is run through both the full RAG system and a vanilla GPT-3.5 baseline (no retrieval). Responses are scored 1–5 by a separate GPT-as-judge call evaluating relevance and groundedness.

```
Irrelevant = score ≤ 2
Baseline:  10/20 queries scored irrelevant
RAG:        6/20 queries scored irrelevant
Reduction: 40%
```

See [`evaluation/benchmark_chart.png`](evaluation/benchmark_chart.png) for the full per-query breakdown.
